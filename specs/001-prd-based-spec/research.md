# Phase 0 Research: PRD 기반 SoC 아키텍처 탐색 및 PPA 추정

## Decision 1: 탐색 실행은 결정적(Deterministic) 파이프라인으로 고정
- Decision: 동일 입력 식별값 + 동일 버전 조합에서 후보 구성/순위/PPA 수치를 완전히 동일하게 반환한다.
- Rationale: 스펙의 SC-007(불일치율 0%)와 FR-009(오차 0)를 만족하려면 랜덤성/비결정성을 제거해야 한다.
- Alternatives considered:
  - 확률 기반 탐색 허용: 성능은 유리할 수 있으나 재현성 요구에 위배.
  - 수치 허용 오차 모델: 사용자 확정 답변(오차 0)과 충돌.

## Decision 2: 상위 5개 후보를 기본 UX/리포트 계약으로 채택
- Decision: 탐색 엔진은 최대 200개까지 계산하되 화면 및 기본 리포트에는 상위 5개만 제공한다.
- Rationale: 사용자의 명시 답변(상위 5개)을 반영하고, 의사결정 집중도와 응답 안정성을 높인다.
- Alternatives considered:
  - 전체 후보 노출: 정보 과부하 및 렌더링 비용 증가.
  - 상위 10/20개 고정: 사용자 결정과 불일치.

## Decision 3: 실패 처리 정책은 자동 재시도 3회
- Decision: 탐색 실패 시 동일 요청에 대해 자동 재시도 최대 3회를 수행하고 실패 사유 코드를 반환한다.
- Rationale: 일시적 오류 복구율을 높이면서도 무한 재시도/지연 누적을 방지한다.
- Alternatives considered:
  - 재시도 없음: 일시적 오류에 취약.
  - 2회 재시도: 복구율이 낮을 가능성.

## Decision 4: 카탈로그 동시 수정은 충돌 감지 후 저장 차단
- Decision: 동일 IP 항목 동시 수정 시 자동 덮어쓰기를 금지하고 수동 병합 워크플로우를 강제한다.
- Rationale: 잘못된 카탈로그 갱신은 모든 탐색 결과를 오염시킬 수 있어 무결성 우선 정책이 필요하다.
- Alternatives considered:
  - Last write wins: 데이터 유실 위험.
  - 자동 병합: 도메인 의미 충돌을 숨길 위험.

## Decision 5: 계약 중심 검증(Contract-First) 적용
- Decision: 탐색 요청/결과/리포트/카탈로그 갱신 인터페이스를 OpenAPI 계약으로 고정하고,
  테스트에서 계약 검증을 의무화한다.
- Rationale: 헌법의 테스트 증거/품질 게이트를 충족하고 팀 간 해석 차이를 줄인다.
- Alternatives considered:
  - 코드 기준 문서화: 문서-구현 불일치 리스크 증가.
  - 화면 중심 수동 검증: 회귀 검출력이 낮음.

## Decision 6: 성능 검증은 부하 + 정확도 이중 축으로 운영
- Decision: 응답시간/실패율(운영 품질)과 정확도 지표(도메인 품질)를 함께 릴리스 게이트로 사용한다.
- Rationale: 사용자 체감 성능과 PPA 품질을 동시에 만족해야 제품 가치가 성립한다.
- Alternatives considered:
  - 응답속도만 측정: 잘못된 추천이 통과될 수 있음.
  - 정확도만 측정: 실사용 대기시간 리스크 방치.

## Resolved Technical Context Unknowns

초기 검토에서 불확실했던 실행 정책(재현성, 노출 개수, 재시도, 충돌 처리)은 모두 사용자 확정
답변과 위 의사결정으로 해소되었으며, 미해결 질의는 없다.
