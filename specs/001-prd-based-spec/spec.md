## Clarifications

### Session 2026-03-04

- Q: 제품 기획자 리포트의 기밀정보 마스킹 기준은 무엇인가? → A: 제품 기획자 리포트에 모든 수치 공개 (마스킹 없음)
- Q: 사용자에게 보여줄 후보 개수 정책은 무엇인가? → A: 상위 5개
- Q: 동일 입력 재실행 시 결과 재현 기준은 무엇인가? → A: 완전 동일(오차 0)
- Q: 탐색 실패 시 자동 재시도 횟수는 얼마인가? → A: 3회
- Q: IP 카탈로그 동시 수정 충돌은 어떻게 처리하는가? → A: 충돌 감지 후 저장 차단, 수동 병합

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 고객 제안용 아키텍처 도출 (Priority: P1)

제품 기획자는 고객 요구사항을 입력하고, 비교 가능한 SoC 아키텍처 후보와 PPA 요약을
얻어 고객 제안안을 빠르게 준비한다.

**Why this priority**: 고객 제안 리드타임 단축이 초기 비즈니스 가치에 직접 연결되며, 본 제품의
핵심 가치를 가장 먼저 검증할 수 있다.

**Independent Test**: 제품 기획자가 단일 요구사항 세트를 입력해 후보 목록, 비교 결과,
내보내기 가능한 제안 리포트를 획득하면 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 제품 기획자가 전력/면적/메모리 요구사항을 입력한 상태, **When** 후보 생성을 실행하면,
   **Then** 유효한 아키텍처 후보 목록과 각 후보의 PPA 요약이 제공된다.
2. **Given** 후보 목록이 생성된 상태, **When** 상위 후보를 선택해 리포트를 내보내면,
   **Then** 고객 공유 가능한 제안 리포트가 생성된다.

---

### User Story 2 - 시스템 아키텍트의 트레이드오프 탐색 (Priority: P2)

시스템 아키텍트는 아키텍처 파라미터를 조정하며 성능/전력/면적의 상충 관계를 탐색해
최적 설계 후보를 선별한다.

**Why this priority**: 정확한 의사결정을 위해 반복 탐색이 필수이며, 제품의 전문 사용자 만족도를
좌우한다.

**Independent Test**: 시스템 아키텍트가 동일한 요구사항에서 파라미터를 변경해 재탐색하고,
파레토 비교 결과로 후보 우선순위를 재정렬할 수 있으면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 시스템 아키텍트가 기준 요구사항으로 초기 결과를 확보한 상태, **When** 메모리 및
   대역폭 관련 파라미터를 변경해 재실행하면, **Then** 변경된 조건에 맞는 새로운 후보 비교 결과가
   제공된다.
2. **Given** 다수 후보가 존재하는 상태, **When** 파레토 기반 비교를 수행하면,
   **Then** 지배/비지배 관계를 기반으로 후보 우선순위를 확인할 수 있다.

---

### User Story 3 - IP 오너의 카탈로그/모델 최신화 (Priority: P3)

IP 오너는 신규 IP 버전과 파라미터 변형 및 PPA 모델 버전을 반영해 탐색 결과의 최신성과
재현성을 유지한다.

**Why this priority**: 정확도와 신뢰성을 장기적으로 유지하기 위한 운영 필수 기능이지만, 초기 MVP
가치 창출은 P1/P2가 더 직접적이다.

**Independent Test**: IP 오너가 신규 IP/모델 버전을 등록한 뒤 동일 요구사항 재실행 시 결과 버전
메타데이터가 갱신되어 추적되면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** IP 오너가 신규 IP 버전 정보를 보유한 상태, **When** 카탈로그를 갱신하면,
   **Then** 이후 생성되는 후보는 최신 카탈로그를 기준으로 산출된다.
2. **Given** 모델 버전이 갱신된 상태, **When** 동일 입력으로 재실행하면,
   **Then** 결과 스냅샷에 모델/룰셋/입력 식별 정보가 함께 기록되어 재현 가능성이 보장된다.

---

### Edge Cases

- 입력 예산(전력/면적)과 필수 성능 목표가 동시에 충족 불가능한 경우, 시스템은 실패가 아닌
  "충족 불가" 결과와 완화 제안을 제공해야 한다.
- 후보 생성 수가 최대치(200개)에 도달한 경우에도 사용자 화면과 기본 리포트에는 상위 5개를 우선순위 기준으로 안정적으로 제공해야 한다.
- 특정 세그먼트에서 유효한 IP 조합이 없는 경우, 원인(호환성/제약 초과)을 사용자에게 명시해야 한다.
- 동일 입력과 동일 버전 조합에서 재실행한 결과가 단 하나라도 다를 경우 실패로 처리하고, 불일치 원인을 즉시 표시해야 한다.
- 권한 수준과 관계없이 리포트 내 수치 정보는 동일하게 제공되어야 하며, 접근 거부보다는 조회 이력을 남겨야 한다.
- 탐색 실패 시 시스템은 최대 3회 자동 재시도 후 최종 실패 사유 코드를 사용자에게 제공해야 한다.
- 동일 IP 항목을 둘 이상의 사용자가 수정한 경우 시스템은 충돌을 감지해 저장을 차단하고 수동 병합을 요구해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용자 요구사항(공정, 전력 예산, 면적 예산, 메모리 구성, 워크로드 특성)을
  입력받아 탐색 가능한 형식으로 정규화해야 한다.
- **FR-002**: 시스템은 정규화된 요구사항을 기반으로 호환 가능한 아키텍처 후보를 자동 생성해야 한다.
- **FR-003**: 시스템은 유효하지 않은 조합을 자동 제외하고 제외 사유를 확인 가능하게 제공해야 한다.
- **FR-004**: 시스템은 각 후보에 대해 성능, 전력, 면적 추정 결과를 함께 제공해야 한다.
- **FR-005**: 사용자는 후보 간 비교를 통해 상위 대안을 선택할 수 있어야 한다.
- **FR-005a**: 시스템은 생성된 후보가 5개를 초과할 경우, 사용자 화면과 기본 리포트에 상위 5개 후보를 기본 제공해야 한다.
- **FR-006**: 시스템은 파레토 기반 비교 결과를 제공하여 트레이드오프 의사결정을 지원해야 한다.
- **FR-007**: 시스템은 고객 공유용 아키텍처 리포트를 내보낼 수 있어야 한다.
- **FR-008**: IP 오너는 신규 IP 버전, 파라미터 변형, 모델 버전을 등록/갱신할 수 있어야 한다.
- **FR-008a**: 동일 IP 카탈로그 항목 동시 수정 시 시스템은 충돌을 감지하고 자동 덮어쓰기를 금지해야 한다.
- **FR-009**: 시스템은 동일 입력 식별값과 동일 버전 조합에서 재실행 시 후보 구성, 순위, PPA 수치가 완전히 동일해야 하며(오차 0), 결과 스냅샷에 검증 근거 메타데이터를 기록해야 한다.
- **FR-010**: 시스템은 역할별 접근권한에 따라 기능 및 데이터 노출 범위를 제어해야 한다.
- **FR-011**: 제품 기획자에게 제공되는 리포트는 시스템 아키텍트와 동일한 수치 정보(절대값 포함)를 제공해야 한다.
- **FR-012**: 시스템은 탐색 실패 시 재시도 여부와 실패 사유를 사용자에게 명확히 전달해야 한다.
- **FR-012a**: 시스템은 탐색 실패 시 자동으로 최대 3회 재시도해야 하며, 모든 재시도 실패 시 최종 실패 사유 코드를 반환해야 한다.

### Key Entities *(include if feature involves data)*

- **Requirement Profile**: 탐색 입력 조건 묶음(전력/면적/메모리/워크로드 등)과 정규화 결과.
- **IP Catalog Item**: 후보 생성에 사용되는 IP 블록/버전/파라미터 변형 정보.
- **Architecture Candidate**: 생성된 SoC 구성안과 유효성 상태 및 제외 사유.
- **PPA Estimate**: 후보별 성능/전력/면적 추정 결과와 비교 지표.
- **Result Snapshot**: 실행 시점의 입력 식별값, 모델/룰셋/카탈로그 버전, 결과 요약.
- **Proposal Report**: 사용자 권한에 맞게 정리된 아키텍처 제안 결과물.
- **Role Policy**: 시스템 아키텍트/제품 기획자/IP 오너의 접근 및 공개 규칙.

## User Experience Consistency *(mandatory for user-facing changes)*

- **UX-001**: 입력, 후보 탐색, 비교, 리포트 내보내기 흐름은 모든 시장 세그먼트에서 동일한 단계 구조를
  유지해야 한다.
- **UX-002**: 모든 주요 화면은 로딩/빈 결과/오류/성공 상태를 명시적으로 구분해 동일한 방식으로 안내해야 한다.
- **UX-003**: 동일 개념(예: 후보, 파레토, 예산 초과)은 화면/리포트 전반에서 동일 용어를 사용해야 한다.
- **UX-004**: 오류 메시지는 원인과 다음 행동(입력 수정, 재시도, 권한 요청)을 함께 제공해야 한다.

## Performance Requirements *(mandatory for runtime-impacting changes)*

- **PRF-001**: 최대 200개 후보 생성 조건에서 사용자 요청의 p95 완료시간은 10초 이하여야 한다.
- **PRF-002**: 동일 조건에서 p99 완료시간은 20초 이하여야 한다.
- **PRF-003**: 동시 사용자 20명 기준 탐색 요청 실패율은 1% 이하여야 한다.
- **PRF-004**: 월간 서비스 가용성은 계획 점검 시간을 제외하고 99.5% 이상이어야 한다.
- **PRF-005**: 결과 정확도 검증에서 PPA 순위 정확도 90% 이상,
  전력 추정 오차 15% 이하, 면적 추정 오차 10% 이하를 달성해야 한다.

## Assumptions

- 초기 릴리스 범위는 아키텍처 초기 탐색 단계이며, 상세 물리 설계/신호 무결성/열 연동 해석은 제외한다.
- 성공 지표 평가는 시장 세그먼트별 사전 정의 평가 데이터셋으로 수행한다.
- 역할은 시스템 아키텍트, 제품 기획자, IP 오너의 3가지로 시작하며, 추가 역할은 후속 범위로 본다.
- 보고서 소비자는 기술 상세보다 의사결정 지표를 우선적으로 요구한다고 가정한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 제품 기획자는 단일 요구사항 세트 입력 후 10분 이내에 고객 공유 가능한 제안 리포트를
  생성할 수 있다.
- **SC-002**: 탐색 요청의 95%는 10초 이내, 99%는 20초 이내에 결과를 제공한다.
- **SC-003**: 각 시장 세그먼트별로 PPA 순위 정확도 90% 이상을 달성한다.
- **SC-004**: 각 시장 세그먼트별로 전력/면적 추정 오차가 각각 15%/10% 이하를 유지한다.
- **SC-005**: 평가 기간 동안 리포트 권한 정책 위반(비인가 사용자 접근) 발생률 0%를 유지한다.
- **SC-006**: 주요 사용자(시스템 아키텍트/제품 기획자) 대상 시나리오 테스트에서 핵심 과업 완료율 90% 이상을
  달성한다.
- **SC-007**: 동일 입력 식별값과 동일 버전 조합으로 100회 반복 실행 시 결과 불일치율 0%를 달성한다.
- **SC-008**: 일시적 실패가 포함된 탐색 요청에서 자동 재시도(최대 3회) 후 최종 성공 전환율 95% 이상을 달성한다.
- **SC-009**: 동시 수정 충돌 시험에서 자동 덮어쓰기 발생률 0%를 달성한다.
